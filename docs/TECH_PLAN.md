# 기술 계획서 — 전체 검토와 구현 순서

이 문서는 프로젝트 전체의 기술 계획을 검토하고 구현 순서를 정한 최상위 문서다. 클로드 코드는 큰 방향을 판단할 때 이 문서를 기준으로 삼는다. 세부 작업은 docs 폴더의 PROJECT_OVERVIEW.md, BACKEND_SPEC.md, WEBSITE_SPEC.md, APP_SPEC.md를 따른다.

## 1. 우리가 만드는 것 (한 줄)

시각장애인용 웨어러블 안내 AI. 안경 카메라로 눈앞을 보고, 사용자의 시각 상태(선천맹, 중도실명, 저시력)에 맞춰 생생한 한국어로 설명해준다.

핵심 차별점은 개인화다. 같은 대상도 시각 상태에 따라 다르게 설명한다. 특히 색. 선천맹에게는 색을 시각 기억으로 설명하지 않고 온도나 질감이나 문화적 의미로 짧게 풀고, 중도실명에게는 색 표현을 그대로 쓴다. 이 개인화가 제품의 심장이고 특허 후보다.

## 2. 전체 시스템 구성

세 덩어리가 하나의 데이터 구조를 공유한다.

- 백엔드. 실시간으로 이미지를 인식하고 맞춤 설명을 만드는 뇌. 데이터 저장소를 관리한다. 웹사이트와 앱이 공통으로 부른다.
- 웹사이트. 회사 소개와 서비스 안내(공개), 작가가 작품 정보를 넣는 입력 페이지(나중에 로그인으로 분리). 넣은 작품은 백엔드에 저장된다.
- 앱. 안경과 연결되어 시각장애인이 쓴다. 안경 카메라 이미지를 백엔드로 보내고 설명을 음성으로 들려준다.

폴더 구조.

```
blind-guide/
  backend/   # 백엔드
  website/   # 웹사이트
  app/       # 앱
  shared/    # 공유 데이터 구조
  docs/      # 지시서 문서들
```

## 3. 백엔드 상세 (기술 중심)

### 3.1 처리 흐름

앱 요청 → API 입구(FastAPI) → 인식(이미지를 밋밋한 묘사로) → 작품/프로필 조회 → 리라이터(시각 상태에 맞춘 설명 생성) → 응답

### 3.2 인식 계층

- 역할. 이미지를 받아 밋밋한 묘사를 만든다. 나중에는 어느 작품인지 식별까지.
- 기술 방침. 우리가 인식 모델을 직접 만들지 않는다. 오픈소스 VLM을 가져다 쓴다. 여기 기술 투자할 여력이 없고, 우리 투자처는 리라이터 하나다.
- 뼈대 단계. 가짜 함수로 둔다. 이미지를 무시하고 고정 묘사를 반환. 함수 경계를 지켜서 나중에 VLM으로 교체할 때 다른 코드를 안 건드리게 한다.
- 다음 단계. 오픈소스 VLM을 붙여 실제 묘사를 뽑는다. 작품 식별은 그 다음. 이미지 식별은 각도와 조명 때문에 어려우므로, 나중에 QR이나 NFC 태깅으로 정확도를 보강한다. 지금은 이미지 우선, 태깅은 보강이라는 순서만 기억한다.

### 3.3 리라이터 계층 (우리 핵심)

- 역할. 밋밋한 묘사 + 사용자 프로필 + 작품 정보를 받아, 시각 상태에 맞춘 생생한 한국어 설명을 만든다. 글에서 글로. 이미지는 안 본다.
- 뼈대 단계. 외부 LLM API에 우리 시스템 프롬프트를 얹어 리라이터 역할을 시킨다. 프롬프트에는 화면해설 원칙(객관 묘사, 공간 순서, 간결, 지어내지 않기, 구어체)과 시각 상태별 색 규칙이 들어간다. 원문은 BACKEND_SPEC에 있다.
- LLM 호출부는 반드시 별도 함수로 분리한다. 이 함수만 교체하면 API에서 자체 모델로 갈아탈 수 있어야 한다.

### 3.4 자체 LLM 모델 계획 (리라이터의 최종 형태)

이 부분은 코드보다 데이터가 먼저다. 순서를 지킨다.

1. 평가셋부터 만든다. 밋밋한 묘사와 정답 설명의 쌍. 학습에 안 쓰는 것으로 따로 빼둔다. 이게 있어야 무엇이 나아졌는지 잰다.
2. 프롬프트와 컨텍스트로 최대한 뽑는다. 맞춤형의 대부분은 학습이 아니라 프롬프트로 풀린다. 이전에 범용 API로 자연스러운 해설은 나왔는데 맞춤형이 아니었던 것은 프롬프트에 시각장애 맥락을 안 준 탓이 컸다.
3. 그래도 남는 격차만 파인튜닝한다. 격차는 주로 톤, 색을 교차감각으로 푸는 방식, 한국어 화면해설 관례다.

파인튜닝 방침.
- 베이스는 한국어 오픈소스 텍스트 모델. 후보는 EXAONE, HyperCLOVA X SEED, Qwen3, Gemma. 크기는 3B에서 8B면 충분하다. 리라이터는 무거울 필요 없다.
- 방법은 LoRA 또는 QLoRA. A100 한 장이면 된다. 도구는 Unsloth나 LLaMA-Factory.
- 학습 데이터는 증류로 만든다. 강한 AI에 원칙만 주고(대본 입력 금지) 정답을 생성시키고, 사람이 감수한다. 양보다 질. 잘 만든 수백 개가 우선.
- 데이터 형식. profile(onset, interest, length), raw(밋밋한 묘사), target(정답 설명).
- 저작권 규칙. 방송 화면해설 대본을 긁어 학습하지 않는다. 생성 AI에 대본을 입력하지도 않는다. 원칙과 방법론 참고는 자유. 정식 협력으로 허락받으면 그때 대본 학습 가능.

완성되면 3.3의 LLM 호출 함수만 자체 모델 호출로 교체한다. 백엔드 나머지는 그대로.

검토 체계. 이 자체 모델 파트는 단계마다 구현 전에 별도 검토 창(설계 담당 클로드)에서 검토받고 진행한다. 특히 다음 지점들은 코드 작업 전에 반드시 검토를 거친다. 평가셋 형식과 예시, 증류 프롬프트 설계, 프롬프트만으로 어디까지 풀리는지 판단, 파인튜닝 착수 여부와 베이스 모델 선정, 학습 데이터의 저작권 검토. 클로드 코드는 이 지점들에서 임의로 진행하지 말고 검토가 끝난 결정을 받아 구현한다.

### 3.5 저장소 계층

- 작품(Artwork). id, image_url, title, artist, material, size, year, description(작가가 쓴 글, 선택).
- 사용자 프로필(UserProfile). user_id, onset(congenital/acquired/unknown), interest, length.
- 뼈대 단계. 코드 안 예시 데이터. 나중에 실제 DB로 교체할 자리를 표시해둔다.
- 웹사이트가 저장하는 작품과 앱이 꺼내 쓰는 작품이 같은 저장소를 본다.

### 3.6 배포와 확장 (나중 단계, 방향만)

- 현장(행사장, 미술관)에는 로컬 서버(젯슨 또는 GPU 박스)를 둔다. 전용 공유기를 가져가면 인터넷 없이 자기완결로 돈다. 지연도 줄고 데이터도 우리가 갖는다.
- 규모가 커지면 vLLM 배치 처리, 자주 쓰는 설명의 사전 계산 캐시, GPU 수평 확장 순으로 감당한다. 감으로 맹학교 수십 명은 젯슨 한 대, 수백 명 동시는 A100 한두 장.
- 뼈대 단계에서는 신경 쓰지 않는다. 로컬 개발 서버로 충분하다.

## 4. 웹사이트 상세

- 공개 영역. 홈(회사 소개)과 서비스 페이지. 지금은 최소 디자인.
- 입력 영역. 작품 등록 폼과 작품 목록. 폼 항목은 사진(필수), 제목, 작가명, 재질, 크기, 제작연도, 설명 글(선택). id는 자동 부여.
- 등록하면 백엔드의 작품 저장 엔드포인트(POST /artworks)로 보낸다. 이 엔드포인트는 웹사이트 작업 때 백엔드에 추가한다.
- 공개와 입력을 라우트로 분리해 나중에 입력 쪽만 로그인을 걸 수 있게 한다. 로그인 자체는 이번에 안 만든다.

## 5. 앱 상세

- 최종 형태. 안경 카메라 이미지를 백엔드로 보내고 설명을 음성으로 출력. 안경은 메타 안경, 폰이 릴레이.
- 뼈대 단계. 안경 없이 간다. 화면에 버튼 하나, 누르면 백엔드 POST /describe를 호출하고 온 설명을 글자로 표시. 프로필은 앱 안 예시 값. 음성 없음.
- 안경 SDK 연동은 백엔드와 웹사이트가 자리 잡은 뒤. 메타의 공식 SDK와 문서가 있으므로 그때 붙인다.

## 6. 구현 순서 (전체 로드맵 — A100 마감 기준 압축 일정)

제약. 메가존 A100은 이미 카운트 중이며 8월 중순까지 약 한 달 반 남았다. 월 200만원어치, 두 달 중 잔여분. 모든 일정은 이 마감에 맞춘다. A100은 학습 돌릴 때만 켜고 끝나면 바로 끈다. 시험과 프롬프트 작업은 전부 데스크탑(RTX 5070)에서 한다.

1주차 (7/3~7/13). 뼈대 완성.
- 백엔드 관 뚫기. 요청이 들어와 리라이터(API)를 거쳐 맞춤 설명이 나가는 것. 이번 주말까지가 최우선 마감.
- 웹사이트 뼈대(작품 등록과 목록), 앱 뼈대(버튼 누르면 설명 표시).
- LLM 창. 데스크탑에 로컬 모델 띄우고 평가셋 초안 20개.
- A100 첫 접속과 환경 연습을 이번 주에 미리 해둔다(반나절).

2주차 (7/14~7/20). 데이터.
- 평가셋 확정(검토 창 검토 후). 증류로 학습 데이터 300~500개 생산, 샘플 감수.
- 인식(VLM)을 백엔드에 실제 연결.
- 마감. 7/20까지 학습 데이터 300개.

3주차 (7/21~7/27). 첫 학습.
- 베이스 모델 확정(검토 창에서, 데스크탑 시험 결과 기반. 현재 우선순위 EXAONE).
- A100에서 첫 LoRA 학습 → 평가셋으로 측정 → 부족한 점 파악. 한 바퀴 완주가 목표.
- 마감. 7/27까지 첫 학습 사이클 완료.

4~5주차 (7/28~8/10). 반복 개선.
- 데이터 보강 → 재학습 → 평가 사이클을 두세 번. A100 시간을 여기에 집중 투입.

마지막 주 (8/11~마감). 마무리.
- 최적 모델 선정, 백엔드 LLM 호출 함수를 자체 모델로 교체, 전체 연동 확인.
- A100 잔여 시간으로 파일럿 서빙 시험.

이 기간에 하지 않는 것. 웹사이트 디자인 다듬기, 앱 음성 출력과 안경 SDK 연동, 위치 태깅. 전부 A100 마감과 무관하므로 그 뒤로 미룬다.

## 7. 공통 규칙

- API 키와 비밀값은 환경변수로만. 하드코딩 금지.
- 각 덩어리는 자기 폴더에만. 함수 경계를 지켜 교체 가능하게.
- 공유 데이터 구조(Artwork, UserProfile)를 어기지 않는다.
- 한 번에 다 만들지 않는다. 관을 먼저 뚫고 조각을 하나씩 진짜로 바꾼다.
