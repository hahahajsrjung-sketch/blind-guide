# 웹사이트 작업 지시서

PROJECT_OVERVIEW.md를 먼저 읽고 이 문서를 본다. 작품 저장은 그 문서의 Artwork 공유 구조를 따른다.

## 1. 웹사이트가 하는 일

두 가지를 한 사이트에 담는다. 지금은 같이 두되, 나중에 나눌 수 있는 구조로 만든다.

- 공개 부분: 회사 소개와 서비스 내용. 누구나 본다.
- 입력 부분: 작가가 작품 정보를 넣는 페이지. 넣은 작품은 백엔드를 통해 저장된다.

## 2. 이번 작업 범위 (뼈대)

- 회사 소개와 서비스 내용을 담은 공개 페이지.
- 작가가 작품을 입력하는 페이지. 입력하면 Artwork 구조로 백엔드에 저장.
- 저장된 작품 목록을 보여주는 간단한 페이지.

가짜로 두거나 나중에 할 것:
- 로그인. 지금은 없이 입력 페이지를 그냥 연다. 나중에 막을 자리만 구조로 분리.
- 실제 데이터베이스. 백엔드의 저장소에 저장(지금은 예시 데이터와 같은 자리).
- 세련된 디자인. 뼈대 단계에선 최소한으로.

## 3. 페이지 구조

```
website/
  공개 페이지
    홈: 회사 소개, 우리가 뭘 하는지 한눈에.
    서비스: 서비스 내용 설명. 시각장애인을 위한 안내 AI가 무엇인지.
  입력 페이지 (나중에 로그인으로 막을 영역)
    작품 등록: 작가가 작품 정보를 넣는 폼.
    작품 목록: 등록된 작품을 확인.
```

공개 영역과 입력 영역을 폴더나 라우트로 분리해서, 나중에 입력 영역에만 로그인을 걸 수 있게 한다.

## 4. 작품 등록 폼에서 받는 것

PROJECT_OVERVIEW의 Artwork 구조에 맞춘다.
- 작품 사진 업로드. 필수. 이미지 인식의 기준이자 설명 재료.
- 제목.
- 작가명.
- 재질.
- 크기.
- 제작연도.
- 설명 글. 선택. 작가가 원하면 직접 쓴다. 비워도 됨.

id는 시스템이 자동 부여하므로 폼에 없다.

폼을 제출하면 이 값들이 Artwork 구조로 백엔드에 저장된다. 백엔드에 저장 요청을 보내는 부분을 함수로 분리한다.

## 5. 백엔드와의 연결

웹사이트는 작품을 직접 저장하지 않고 백엔드에 저장을 요청한다. 백엔드에 작품 저장용 엔드포인트가 필요하다(예: POST /artworks). 이 엔드포인트는 BACKEND_SPEC에 없으니, 웹사이트 작업 시 백엔드에 이 저장 엔드포인트를 추가한다. 저장되는 형식은 Artwork 구조.

즉 흐름은 이렇다. 작가가 폼 제출 → 웹사이트가 백엔드 POST /artworks 호출 → 백엔드가 저장소에 Artwork로 저장. 나중에 앱 사용자가 그 작품을 비추면 같은 저장소에서 꺼내 쓴다.

## 6. 기술 선택

- 뼈대 단계라 가벼운 걸로 시작한다. 복잡한 프레임워크보다, 먼저 페이지가 뜨고 폼이 백엔드로 값을 보내는 것에 집중한다.
- 사진 업로드는 파일을 백엔드로 보내 저장하고, 저장된 위치를 image_url로 갖는다. 이번엔 로컬 폴더에 저장해도 된다.

## 7. 완료 기준

브라우저에서 홈과 서비스 페이지가 뜬다. 작품 등록 페이지에서 사진과 정보를 넣고 제출하면 백엔드에 Artwork로 저장되고, 작품 목록 페이지에서 방금 넣은 작품이 보인다. 이 저장된 작품을 백엔드가 앱 요청 때 꺼내 쓸 수 있는 상태면 성공.

## 8. 공통 규칙

- 비밀값은 환경변수. 하드코딩 금지.
- 공개 영역과 입력 영역을 구조로 분리해 나중에 로그인 추가가 쉽게.
- Artwork 공유 구조를 어기지 않는다.
