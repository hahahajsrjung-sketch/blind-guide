# 백엔드 작업 지시서

PROJECT_OVERVIEW.md를 먼저 읽고 이 문서를 본다. 여기서 정의한 구조는 그 문서의 공유 데이터 구조를 따른다.

## 1. 백엔드가 하는 일

안경 앱에서 이미지와 사용자 정보가 오면, 이미지를 인식해 밋밋한 묘사를 얻고, 사용자의 시각 상태에 맞춰 다시 써서 설명을 돌려준다.

요청 흐름:
앱 → API 입구 → 인식(이미지를 묘사로) → 작품 정보 조회 → 리라이터(맞춤 설명) → 응답 출구 → 앱

## 2. 이번 작업 범위 (뼈대)

가장 얇은 관 하나를 뚫는다. 요청이 들어와 리라이터를 거쳐 맞춤 설명이 나가는 것.

가짜로 둘 것:
- 인식: 이미지를 받지만 무시하고 고정된 묘사를 반환한다. 실제 VLM은 안 붙인다. 함수로 분리해 나중에 교체.
- 작품 정보와 사용자 프로필: 코드 안 예시 데이터. 나중에 DB로 뺄 자리를 주석 표시.

리라이터는 외부 LLM API로 구현한다. 우리 파인튜닝 모델은 아직 없으니 API에 시스템 프롬프트를 얹어 리라이터 역할을 시킨다. LLM 호출 부분을 별도 함수로 분리해 나중에 로컬 모델로 교체 가능하게 한다.

## 3. 코드 구조

```
backend/
  main.py           # API 입구. FastAPI. 요청 받고 파이프라인 호출하고 응답 반환.
  recognition.py    # 인식. 지금은 가짜 묘사 반환. 나중에 VLM.
  rewriter.py       # 리라이터. 묘사+프로필+작품정보로 프롬프트 조립하고 LLM API 호출.
  store.py          # 작품 정보와 사용자 프로필 저장소. 지금 예시 데이터. 나중에 DB.
  prompts.py        # 시스템 프롬프트와 시각 상태별 규칙.
  config.py         # API 키 등 설정. 환경변수에서 읽음.
```

## 4. 각 조각

### main.py
- FastAPI 앱.
- 엔드포인트: POST /describe
- 요청 바디: user_id(문자열), image(이번엔 실제 파일 대신 아무 값이나 받아도 됨. 자리만), artwork_id(문자열, 선택. 어느 작품인지 알 때).
- 처리: store에서 user_id로 프로필 조회, artwork_id로 작품 조회, recognition으로 묘사 얻기(가짜), rewriter로 설명 생성, JSON 반환.
- 주석: 나중에 image로 recognition이 작품을 식별하도록 확장할 자리 표시.

### recognition.py
- 함수: recognize(image) -> str
- 지금은 image 무시하고 고정 묘사 반환. 예: "받침대 위에 선 남자 조각상. 대리석."
- 나중에 VLM으로 이미지를 묘사로 바꾸고, 나아가 어느 작품인지 식별하도록 교체. 함수 모양은 유지.

### rewriter.py
- 함수: rewrite(raw_description, profile, artwork) -> str
- prompts에서 시스템 프롬프트와 시각 상태별 규칙을 가져온다.
- profile의 onset에 따라 색 설명 규칙 선택.
- artwork 정보(제목, 작가, 재질 등)가 있으면 컨텍스트로 프롬프트에 넣는다. artwork의 description(작가가 쓴 글)이 있으면 그걸 우선 참고 재료로 쓴다.
- 조립해서 call_llm 호출하고 결과 반환.
- call_llm은 별도 함수. 나중에 로컬 모델로 교체할 부분.

### store.py
- 작품 딕셔너리. artwork_id를 키로, 값은 PROJECT_OVERVIEW의 Artwork 구조(id, image_url, title, artist, material, size, year, description).
- 사용자 프로필 딕셔너리. user_id를 키로, 값은 UserProfile 구조(onset, interest, length).
- 조회 함수: get_artwork(artwork_id), get_profile(user_id).
- 맨 위 주석: 이 예시 데이터는 나중에 실제 DB로 교체. 웹사이트가 저장한 작품이 여기 들어온다.
- 예시 데이터로 작품 두어 개, 사용자 두 명(한 명은 onset congenital, 한 명은 acquired) 넣어서 차이를 확인할 수 있게 한다.

### prompts.py
아래 원문을 그대로 옮긴다.

시스템 프롬프트:
```
당신은 시각장애인을 위한 화면해설 작가입니다. 눈앞의 대상을 소리로만 전달받는 사람에게, 그 대상을 정확하고 생생하게 전달하는 것이 임무입니다.

핵심 원칙:
1. 객관적으로 묘사한다. 해석, 평가, 감정 단정을 넣지 않는다.
2. 공간을 일관된 순서로 훑는다. 전체에서 부분으로, 위에서 아래로, 왼쪽에서 오른쪽으로.
3. 중요한 것부터, 간결하게. 모든 것을 나열하지 않는다.
4. 예술 작품은 예술로 다룬다. 자극적이거나 평면적인 표현을 피한다.
5. 인식 결과에 없는 사실을 지어내지 않는다.
6. 음성으로 읽힐 자연스러운 구어체 문장. 목록이나 기호나 괄호 설명을 넣지 않는다.
7. 안전 관련 정보가 있으면 가장 먼저 말한다.
```

시각 상태별 색 설명 규칙:
- congenital: 색을 시각 기억으로 설명하지 않는다. 꼭 필요할 때만 온도, 질감, 문화적 의미로 짧게 보조하고 과하지 않게.
- acquired: 색과 시각적 표현을 그대로 사용해도 된다.
- unknown: 색 이름은 쓰되 해석은 얹지 않고 중립적으로.

길이 규칙:
- short: 한두 문장.
- medium: 서너 문장.
- long: 네다섯 문장.

### config.py
- LLM API 키를 환경변수에서 읽는다. 하드코딩 금지.
- 쓸 API와 모델 이름도 여기.

## 5. 완료 기준

로컬에서 서버를 띄우고 POST /describe로 user_id와 artwork_id를 보내면 그 사용자의 시각 상태에 맞춘 설명이 JSON으로 돌아온다. 같은 작품이라도 onset이 congenital인 사용자와 acquired인 사용자에게 색 설명이 다르게 나오면 성공.
