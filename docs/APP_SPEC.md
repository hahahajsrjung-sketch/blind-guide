# 앱 작업 지시서

PROJECT_OVERVIEW.md를 먼저 읽고 이 문서를 본다. 백엔드와 주고받는 형식은 그 문서의 공유 데이터 구조를 따른다.

## 1. 앱이 하는 일

시각장애인이 쓰는 앱. 안경 카메라로 눈앞을 비추면, 그 이미지를 백엔드로 보내고, 백엔드가 돌려준 맞춤 설명을 음성으로 들려준다.

최종 흐름:
안경 카메라 → 앱이 이미지 받음 → 백엔드 POST /describe로 전송 → 맞춤 설명 받음 → 음성으로 출력

## 2. 이번 작업 범위 (뼈대)

앱은 셋 중 제일 복잡하다. 안경 SDK, 카메라, 음성이 다 얽혀 있다. 그래서 이번 단계에서는 안경 없이도 되는 선까지만 만든다.

이번에 실제로 할 것:
- 앱 기본 화면 하나. 버튼 하나와 결과 표시 영역.
- 버튼을 누르면 백엔드 POST /describe에 요청을 보내고, 돌아온 설명을 화면에 글자로 보여준다.
- 사용자 프로필(시각 상태)을 앱이 갖고 있다가 요청에 같이 보낸다. 지금은 앱 안에 예시로 박아둔다.

이번에 가짜로 두거나 나중에 할 것:
- 안경 연결과 카메라. 지금은 안 붙인다. 이미지 대신, 요청에 자리만 채워 보낸다. 실제로는 백엔드도 인식을 가짜로 두므로 이미지 내용은 아직 안 중요하다.
- 음성 출력. 지금은 설명을 글자로만 보여준다. 음성은 나중에.
- 실제 안경 SDK 연동은 백엔드와 웹사이트가 자리 잡은 뒤 다음 단계에서.

즉 이번 앱의 목표는, 앱에서 버튼을 누르면 백엔드를 불러 맞춤 설명을 받아 화면에 띄우는 것. 안경도 음성도 아직 없이, 백엔드와 앱이 실제로 통신하는 것만 확인한다.

## 3. 기술 선택

- 앱은 폰에서 돌고 나중에 안경 SDK와 붙어야 한다. 어떤 프레임워크로 만들지는 클로드 코드가 제안하되, 백엔드와 HTTP로 통신할 수 있고 나중에 카메라와 안경 SDK를 붙이기 쉬운 걸로 고른다.
- 이번 단계에선 화면 하나에 버튼과 텍스트 영역만 있으면 된다. 복잡한 UI는 나중에.

## 4. 백엔드와의 연결

- 앱은 백엔드 POST /describe를 호출한다. 요청에 user_id, image(지금은 자리만), artwork_id(선택)를 담는다.
- 응답으로 온 설명 텍스트를 화면에 표시한다.
- 백엔드 주소는 설정값으로 두고 하드코딩하지 않는다. 로컬에서 테스트할 때는 로컬 백엔드 주소를 가리킨다.

## 5. 완료 기준

백엔드를 로컬에서 띄워둔 상태에서, 앱을 실행하고 버튼을 누르면, 백엔드가 돌려준 맞춤 설명이 앱 화면에 글자로 뜬다. 앱 안 프로필의 시각 상태를 바꾸면 설명이 다르게 오는 것을 확인하면 성공.

## 6. 공통 규칙

- 비밀값과 주소는 설정으로. 하드코딩 금지.
- 안경 SDK, 카메라, 음성은 이번에 안 만든다. 나중에 붙일 자리만 분리해둔다.
- 공유 데이터 구조를 어기지 않는다.
