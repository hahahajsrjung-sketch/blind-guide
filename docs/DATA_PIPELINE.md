# 데이터 파이프라인 — 작가에게 무엇을 받고, AI가 어떻게 정리하고, 관람객이 어떻게 인식하나

이 문서는 "작가 등록 → AI 정리 → 관람객 인식 → 맞춤 설명" 전 과정의 데이터 설계다.
PROJECT_OVERVIEW / TECH_PLAN의 하위 문서이며, shared/schema.json 의 Artwork v2 확장이 이 문서를 따른다.

## 0. 왜 개편하나 (기존 파이프라인의 약점)

| 약점 | 결과 |
|---|---|
| 작가에게 사진 **1장**만 받음 | 관람객이 다른 각도·거리·조명에서 찍으면 매칭 실패 |
| 식별이 **텍스트 대조**뿐 (관람객 사진의 묘사 ↔ 카탈로그 글) | 시각적 유사도를 전혀 못 씀. 비슷한 글의 작품끼리 혼동 |
| 촉각·안전·작가 의도를 받을 **필드가 없음** | 시각장애인에게 가장 중요한 정보가 설명에 못 들어감 |
| 등록 시점에 AI가 정리해두지 **않음** | 관람객 요청 때마다 맨바닥에서 생성 → 느리고 얕음 |

## 1. 작가에게 받는 것 (입력 데이터 정의)

### 1.1 사진 — 식별의 재료 (필수)

| 종류 | 몇 장 | 왜 |
|---|---|---|
| 대표 컷 `main` | 1 | 정면, 좋은 조명. 카탈로그 표지이자 임베딩 기준 |
| 각도 컷 `angle` | 2~4 | 좌/우/비스듬. 관람객은 정면에서 안 찍는다 |
| 세부 컷 `detail` | 0~2 | 질감·붓터치·재질. 근접 촬영 관람객 대응 + 설명 재료 |
| 설치 전경 `context` | 1 | 벽·받침대·주변이 함께. 실제 관람 시야와 가장 비슷 |

**촬영 가이드(폼에 안내):** 폰 카메라면 충분. 작품이 프레임의 60% 이상. 각도 컷은 30~45도. 유리 반사 피하기.

### 1.2 영상 — 선택, 사진의 상위호환

작품 주위를 10~20초 천천히 도는 영상 1개. 서버가 **프레임을 균등 추출**(기본 6장)해 각도 컷처럼 쓴다.
사진 여러 장 찍기 귀찮은 작가에게 더 쉬운 입력이다.

### 1.3 텍스트 — 설명의 재료

| 필드 | 필수 | 내용 |
|---|---|---|
| title / artist / material / size / year | 권장 | 기본 정보 (기존과 동일) |
| description | 선택 | 작가가 쓴 소개 글 (기존과 동일) |
| **tactile** | 권장 | 촉각·재질감: "차갑고 매끈함", "울퉁불퉁한 붓자국". 시각 없이 전달 가능한 감각 |
| **safety** | 권장 | 안전·관람 규칙: "만지면 안 됨", "받침대가 낮아 발에 걸릴 수 있음". **설명에서 가장 먼저 읽힘**(프롬프트 원칙 7) |
| **location** | 권장 | 전시 위치: "2전시실 입구 왼쪽 벽". 식별 보조 + 안내 |
| intent | 선택 | 작가 의도 한두 문장. 해석이 아니라 맥락으로만 사용 |

## 2. AI 정리 (인제스천) — 등록 순간 서버가 하는 일

작가가 저장을 누르면 백엔드가 자동으로:

```
사진들(+영상 프레임)
  ├─→ ① 임베딩 생성 (오픈소스 CLIP) ──→ 작품별 벡터 인덱스에 저장
  └─→ ② VLM 캡션 (대표컷 1장) ─┐
작가 텍스트 전부 ────────────────┴─→ ③ ai_profile 생성 (LLM 1회)
```

- **① 임베딩**: 각 이미지 → 512차원 벡터. 오픈소스 `sentence-transformers`의 CLIP(ViT-B/32), 로컬 실행·무료·키 불필요. 한 작품에 벡터 여러 개(각도별)를 저장해 어떤 각도 사진이 와도 가까운 벡터가 잡히게 한다.
- **② VLM 캡션**: 대표컷을 VLM이 객관 묘사 → 작가가 글을 안 썼어도 시각 정보 확보.
- **③ ai_profile**: LLM이 (기본정보+작가글+캡션+촉각+안전+의도)를 **화면해설 재료로 재구성**해 저장:

```json
"ai_profile": {
  "visual_summary":  "구도·형태·색을 공간 순서(전체→부분)로 정리한 3~5문장",
  "tactile_summary": "촉각·재질 중심 1~2문장 (선천맹 설명의 핵심 재료)",
  "safety_notes":    "안전 문구 그대로 (있으면)",
  "key_features":    ["식별에 쓸 두드러진 특징 3~5개"]
}
```

이 정리본은 **한 번 만들어 저장**된다. 관람객 요청 때는 생성이 아니라 조회만 → 빠르고 일관됨.

## 3. 관람객 인식 — 2단 식별

```
관람객 사진
  ├─ 1차: CLIP 임베딩 → 전체 인덱스와 코사인 유사도
  │        best ≥ 0.85 → 확정              (빠름, LLM 호출 0회)
  │        0.70~0.85  → 후보로 2차에 전달
  │        < 0.70     → 후보 없음
  └─ 2차: VLM 묘사 + LLM 카탈로그 대조 (기존 방식, 폴백)
           확신 없으면 → 작품 미상으로 묘사만 설명 (억지로 안 맞춤)
```

- 1차가 잡으면 VLM 묘사는 **설명 재료로만** 쓰이고 식별엔 안 쓰임 → 지연·비용 감소.
- 데모 작품처럼 실제 이미지가 없는 작품은 자동으로 2차만 탄다 (하위호환).
- 임계값(0.85/0.70)은 환경변수로 조정 가능. 나중에 QR/NFC가 붙으면 0차로 그 위에 얹는다(TECH_PLAN 3.2).

## 4. 맞춤 설명 — 정리본을 먹는 리라이터

리라이터 입력이 "밋밋한 묘사 + 기본정보"에서 다음으로 확장:

```
safety_notes  → 무조건 맨 앞 (원칙 7)
raw(관람객 사진의 실제 묘사) → 지금 눈앞의 상태
visual_summary → 작품의 정확한 시각 구성
tactile_summary → onset=congenital일 때 색 대신 쓸 교차감각 재료
description/intent → 작가의 목소리 (우선 참고)
```

onset별 색 규칙·길이 규칙은 기존 그대로 (prompts.py 원문 불변).

## 5. 오픈소스 선택 근거

| 용도 | 선택 | 이유 |
|---|---|---|
| 이미지 임베딩 | **CLIP ViT-B/32** (sentence-transformers) | 로컬·무료·CPU 가능, 각도 변화에 강함, 2줄 API. 나중에 SigLIP/EVA-CLIP로 교체 가능한 함수 경계 유지 |
| 영상 프레임 추출 | **OpenCV** | 표준. 균등 샘플링 6프레임 |
| 벡터 검색 | **NumPy 코사인** (지금) | 작품 수백 개까지는 완전탐색이 더 단순·충분. 수천 개 넘으면 FAISS로 교체 (경계 분리) |
| VLM/LLM | Gemini (현행) → 자체 모델(4단계) | call_llm/_vlm_describe 경계 그대로 |

## 6. 하위호환

- `image_url`(단수)은 유지 — 웹·앱 기존 코드가 안 깨짐. 서버는 `images[0]`을 `image_url`에 미러.
- 기존 필드만 보내는 등록도 동작 (새 필드 전부 선택).
- `/describe` 요청·응답 계약 불변 (앱 수정 불필요).
