# 로컬 모델 시험 기록

이 문서는 LLM_SPEC 2.1의 결과 기록이다. 데스크탑에서 한국어 소형 모델을 띄우고,
백엔드 리라이터의 시스템 프롬프트를 그대로 얹었을 때 (1) 화면해설풍 설명이 나오는지,
(2) 선천맹/중도실명 프로필에서 색 설명이 실제로 갈라지는지, (3) API 대비 품질이 어느
정도인지를 손으로 시험한 소감이다. 파인튜닝 필요성 판단의 근거로 쓴다.

## 1. 환경

- GPU: NVIDIA GeForce RTX 5070 Ti, VRAM 16GB (nvidia-smi 실측). RAM 63GB, CPU Core Ultra 7 265K 20코어.
- Ollama 0.31.1 (winget 설치). GPU 구동. 7.8B(Q4, ~4.8GB)·2.4B(~1.6GB) 모두 VRAM에 여유롭게 올라감.
- 갈림길 없음: 사양이 넉넉해 로컬 추론으로 확정. API 뼈대로 후퇴할 이유 없었음.

## 2. 시험한 모델

LLM_SPEC이 우선하라던 EXAONE 계열로 두 크기를 받아 비교.

- `exaone3.5:7.8b` — 주 시험 모델 (품질 판정용)
- `exaone3.5:2.4b` — 3B급 대조군 (SPEC이 우선 지목한 크기)

Qwen3 소형은 이번엔 안 받음(EXAONE로 판단이 충분히 섬). 필요 시 대조군으로 추가 가능.

## 3. 방법 (충실 재현)

`llm/scripts/try_local.py`. **backend/prompts.py를 그대로 import**하고,
backend/rewriter.py의 user_prompt 조립 로직을 똑같이 복제했다. 유일한 차이는 call_llm 뿐 —
Gemini API 대신 로컬 Ollama `/api/chat`를 부른다. temperature 0.4.
즉 TECH_PLAN 3.3의 "call_llm만 바꾸면 로컬로 교체된다"를 실제로 확인한 것이기도 하다.

시험 케이스 3종(유화·대리석 조각·추상 색면화)에 각각 onset을 congenital/acquired로만
바꿔 색 설명이 갈라지는지 봤다. 원본 출력은 `%TEMP%\exaone_test_out.txt`(7.8b),
`exaone24_test_out.txt`(2.4b)에 저장.

## 4. 완료 기준 대비 결과

| 완료 기준 | 결과 |
|---|---|
| 로컬 모델이 돌고 화면해설풍 설명이 나온다 | **충족.** 유창한 한국어 해설체. |
| 선천맹/중도실명 색 설명이 다르게 나온다 | **부분 충족.** 차이는 나지만 방향이 약하고 핵심 규칙을 어긴다(아래). |
| notes.md 기록 | 이 문서. |

## 5. 소감 — 프롬프트만으로는 핵심 차별점이 안 잡힌다

가장 중요한 발견. **선천맹(congenital) 색 규칙이 지켜지지 않는다.**
규칙은 "색을 시각 기억으로 설명하지 말고, 필요할 때만 온도·질감·문화적 의미로 짧게 보조".
그런데 두 로컬 모델 모두 선천맹 프로필에서도 "파란 원피스", "붉은 커튼", "주황색/파란색"을
그대로 썼다. 온도·질감 보조("차가운 질감", "따뜻한 기운")를 조금 얹을 뿐, 색 이름을 빼지 못한다.

- **차이가 아예 없진 않다.** congenital 쪽이 "차가운 질감", "문화적 맥락" 같은 보조를 더 넣고,
  acquired 쪽이 더 미적·시각적으로 서술하는 경향은 보였다. 그래서 "다르게 나온다"는
  형식상 확인은 된다. 그러나 제품의 심장인 "선천맹에게 색 이름 대신 교차감각"이라는
  핵심 규칙은 프롬프트만으로는 이 로컬 모델들에서 실현되지 않았다.

그 밖에 공통으로 관찰된 규칙 위반:

- **길이 규칙 무시.** medium=서너 문장인데 7~9문장으로 길게 나옴.
- **해석·평가 금지(규칙1) 위반.** "우아한 분위기", "고요하고 평온한 순간", "인간의 본질을
  탐구" 등 감정·의미 단정을 얹음.
- **안전 규칙(7) 오적용.** 안전 정보가 없는데도 "안전하게 감상하시기 바랍니다",
  "안전 측면에서는 …고정되어 있어" 같은 문장을 습관적으로 붙임.
- **말투.** "안녕하세요, 여러분", "설명드리겠습니다:" 등 방송·다수 청중 인사말투.
  웨어러블 1인 사용자에게는 부적합.

### 크기 비교 (7.8b vs 2.4b)

작을수록 규칙 준수가 뚜렷이 나빠진다.

- 7.8b: 문장이 자연스럽고 구조가 안정적. 규칙 위반은 위와 같음.
- 2.4b: 색 이름을 더 노골적으로 사용. 메타 잡음("화면 해설을 통해 …설명드리겠습니다:"),
  환각("고대 그리스 조각", "햇살 가득 찬 하늘"), 반복 오류("자연스럽게 자연스럽게"),
  "안전 측면에서는 …" 억지 부착이 매 출력마다. 실사용엔 7.8b가 최소선.

### API(Gemini) 대비

백엔드 검증 기록으로는 동일 프롬프트에서 Gemini는 onset별 색 규칙을 실제로 지켰다
(congenital=온도/문화 보조, acquired=색 그대로). 로컬 EXAONE는 같은 프롬프트로 그 수준에
못 미친다. **격차의 정체가 TECH_PLAN 3.4가 예측한 그대로다** — 톤, 색을 교차감각으로 푸는
방식, 한국어 화면해설 관례. 즉 이 격차가 파인튜닝의 표적이 된다.

## 6. 정량 측정 — 자동 규칙 평가기 (llm/scripts/eval_local.py)

위 소감을 수치로 바꿨다. evalset.jsonl 15개를 로컬 모델로 재생성하고, 결정론적 규칙 검사
(색 이름 누출·길이 초과·안전문구 오적용·방송 인사말투)로 셌다. LLM-judge 아님, 문자열 규칙만.
**생성은 greedy(temperature 0, seed 0)로 고정해 재현 가능하다** — baseline은 실행마다 흔들리면
안 되므로 손시험(0.4)과 달리 평가는 온도 0. 전체 표와 생성 원문은
`llm/data/eval_local_results_7.8b.md`, `_2.4b.md`. 재생성 `python scripts/eval_local.py <모델>`.

| 지표 | 7.8b | 2.4b |
|---|---|---|
| 선천맹 색 이름 누출 | 4/6 항목 | 5/6 항목 |
| 길이 초과 | 12/15 | 14/15 |
| 안전문구 오적용 | 6/15 | 10/15 |
| 방송 인사말투 | 5/15 | 0/15 |

쌍 비교 — 같은 raw에서 색 이름 사용 수(congenital vs acquired). 규칙대로면 congenital은
0에 가깝고 acquired보다 확연히 적어야 한다.

| raw | 7.8b (cong/acq) | 2.4b (cong/acq) |
|---|---|---|
| 유화 | 6 / 7 | 3 / 3 |
| 사과 | 0 / 1 | 1 / 1 |
| 추상 색면화 | 2 / 2 | 4 / 3 |
| 전시실 공간 | 2 / 2 | 2 / 2 |

> 2026-07-08 감지기 수정: COLOR_RE의 `자주`가 부사("자주 탐구되는")와 동형이라 오탐이 실제
> 발생(2.4b 추상 cong가 5로 과대집계 → 4가 정확). `자주색|자줏빛`으로 좁히고 누락된 `청록`을
> 추가해 재측정한 표가 위. 방향 반전(cong 4 > acq 3) 결론은 불변. 또 하나 — **greedy(temp0,
> seed0)도 Ollama에서 실행 간 ±1 흔들림이 관측된다**(7.8b 길이초과 12↔13, 안전 6↔7).
> baseline 비교는 ±1을 오차로 보고 그 이상의 변화만 효과로 읽는다.

읽는 법:
- **7.8b는 방향을 뒤집지 않는다.** 4쌍 모두 congenital ≤ acquired이고, 사과·유화 2쌍은
  congenital이 더 적다(사과는 0 — 유일하게 규칙에 맞는 사례). 그러나 유화 congenital이 색
  이름 6개로 매우 높아 규칙(≈0)엔 크게 미달. "방향은 안 틀리지만 강도가 못 미친다."
- **2.4b는 방향이 무너진다.** 추상 색면화에서 congenital(5) > acquired(3)로 규칙과 반대이고,
  나머지 3쌍은 동률이라 구분이 사실상 없다.
- 길이·안전 오적용은 2.4b가 확연히 나쁘다(길이 14/15, 안전 오적용 10/15 — 거의 매 출력마다
  불필요한 안전문구를 붙임). 인사말투만 2.4b가 0인데, 그 자리를 메타 잡음·환각이 채워서
  개선으로 볼 수 없다.

결론 보강: 크기를 줄이면 규칙 준수가 무너진다(방향 반전·길이·안전). 프롬프트만으로 갈 때
최소선은 7.8b이고, 그마저 핵심 색 규칙은 강도가 크게 못 미친다. **이 표가 재현 가능한
baseline이다** — 프롬프트 개선/파인튜닝 전후를 같은 스크립트로 다시 재서 효과를 증명한다.

## 7. 판단 근거로 남기는 것 (결정은 검토 창에서)

- 프롬프트만으로 선천맹 색 규칙이 안 잡힌다 → 프롬프트 개선(few-shot 예시 주입, 규칙을
  더 강하게/짧게, 색 이름 금지를 출력 제약으로)을 먼저 끝까지 밀어볼 여지가 크다.
  TECH_PLAN 3.4 2단계(프롬프트로 최대한)를 아직 다 안 써봤다.
- 그래도 남으면 LoRA. 평가셋(evalset.jsonl)이 이 개선의 효과를 재는 자가 된다.
- **아래는 임의로 진행하지 않고 검토받을 지점이다:** 베이스 모델 최종 선정(EXAONE 7.8b 확정
  여부, Qwen3 대조 추가), few-shot/출력제약 프롬프트 재설계, 파인튜닝 착수 판단.

## 8. 재현 방법

```
ollama serve            # 설치 시 자동 구동됨
python llm/scripts/try_local.py                  # 손시험 3케이스, 7.8b
python llm/scripts/try_local.py exaone3.5:2.4b   # 대조군
python llm/scripts/eval_local.py                 # 정량 규칙 평가(15개), 7.8b
python llm/scripts/eval_local.py exaone3.5:2.4b  # 정량 평가 대조군
```
Windows에서 출력 깨지면 `$env:PYTHONUTF8="1"` 후 실행.
평가기 결과는 `llm/data/eval_local_results*.md`.

## 9. 프롬프트 최대화(v2)와 증류 파이프라인 완성 (2026-07-07)

TECH_PLAN 3.4의 2~4단계를 끝까지 밀어 파이프라인을 완성했다. 요지: **프롬프트로 갈 데까지
가서 남는 격차를 수치로 못박고, 그 격차를 메울 학습 파이프라인을 GPU 박스에서 바로 돌 수 있게
갖췄다.** 모두 llm 폴더 안에서만 작업(backend 미변경).

### 9.1 v2 프롬프트 — few-shot + 출력 제약 (scripts/rewriter_v2.py)

baseline(v1, backend 시스템 프롬프트 단독) 대비, 강한 출력 제약 부록(선천맹 색 이름 금지 명시,
문장 수 상한, 인사말/안전 오적용 금지) + few-shot 3개(평가셋과 겹치지 않는 새 예시)를 얹었다.
같은 eval_local.py로 greedy 재측정(`eval_local.py exaone3.5:7.8b v2`):

| 지표 | v1 baseline | v2 |
|---|---|---|
| 선천맹 색 이름 누출 | 4/6 | **4/6 (그대로)** |
| 길이 초과 | 12/15 | **3/15** |
| 안전문구 오적용 | 6/15 | **1/15** |
| 방송 인사말투 | 5/15 | **0/15** |
| 유화쌍 색수 congenital | 6 | **3** |

(v1 수치는 2026-07-08 감지기 수정('자주' 오탐 제거) 후 재측정본. greedy도 실행 간 ±1
흔들림이 있어 — 첫 재실행에서 길이 13/15·안전 7/15가 나왔다가 재측정에서 12/15·6/15로 복귀 —
±1은 오차로 본다. 6절 참조.)

**결론: 프롬프트로 길이·인사말·안전은 완전히 잡힌다. 그러나 핵심인 "선천맹 색→교차감각"은
프롬프트만으로 안 잡힌다** — 누출 '항목 수'는 4/6로 불변, 누출 '양'만 절반(유화 6→3). 즉 모델이
색 이름을 완전히 빼는 데까지는 못 가고 줄이기만 한다. TECH_PLAN 3.4-3의 "남는 격차=색을
교차감각으로 푸는 방식" 예측을 수치로 확인. → 파인튜닝 정당화.

### 9.2 증류 학습 파이프라인 (scripts/gen_sft_data.py, rule_repair.py, build_chat_dataset.py)

TECH_PLAN "강한 AI에 원칙만 주고 정답 생성, 사람이 감수, 양보다 질"을 코드로.

- **raw 풀** `data/sft/raw_pool.jsonl` 36개 — 미술·공간·일상 혼합, 안전 단서 포함, 색 다양.
  평가셋과 분리(홀드아웃 오염 방지, 생성기가 겹치는 raw 자동 제외).
- **교사 2종.** `local-repair`(EXAONE 7.8b v2 초안 + 결정론적 rule_repair, 지금 자체완결로 돎)
  와 `gemini`(정석 '강한 AI', 키 필요). rule_repair는 관형형 색을 제거("파란 원피스"→"원피스"),
  서술형 색 잔존은 자동 제거가 위험하므로 needs_human 플래그로 감수 큐에 보낸다. 길이 절단·
  인사말/안전 오적용 제거도 결정론적으로.
- **실측 생성(local-repair, medium).** 108개 생성(onset 36/36/36 균형), 자동통과 100 / 감수필요 8.
  **핵심 검증: 자동통과된 선천맹 28개 중 색 잔존 0개**(eval_local의 색 감지기로 확인). 색이 남은
  8개는 정확히 needs_human으로 걸러짐. 자동통과분의 색 치환이 실제로 교차감각으로 갔다
  (예: 노란/검은 세로줄 → "밝은 톤/어두운 톤", 분홍 저고리·남색 치마 → 색 제거). 이게 파인튜닝이
  배울 바로 그 신호다.
- **감수 게이트.** 생성물은 `data/sft/train.review.jsonl`(감수 대기)로 떨어진다. 사람이 needs_human
  우선 감수해 통과분만 `train.jsonl`로 확정(TECH_PLAN "사람이 감수"). meta.draft에 교사 원 초안
  보존 → 감수자가 비교 가능.
- **학습 형식화.** `build_chat_dataset.py`가 확정본을 채팅 형식으로. **시스템 프롬프트는 backend
  원본(최소)** — v2 제약은 안 넣는다. 목적은 v2가 프롬프트로 시키던 규칙을 가중치에 내재화해,
  backend가 프롬프트 변경 없이 모델 태그만 바꾸게 하는 것. 자동통과분 100개로 형식화 실증
  (`finetune/data/train_chat.jsonl`, {system,user,assistant} 100줄).

### 9.3 LoRA 파인튜닝 (finetune/)

`train_lora.py`(Unsloth+TRL) + `config.yaml`(베이스 EXAONE 7.8b, LoRA r16, QLoRA 4bit, 3 epoch,
train_on_responses_only) + `requirements.txt` + `export_to_ollama.md`. 학습은 GPU 박스(메가존 A100).
config의 `base_model` 한 줄로 베이스 교체(Qwen3/Gemma 대조 가능). 학습 후 GGUF로 내보내 Ollama에
`blindguide-rewriter`로 등록 → backend call_llm이 그 태그를 부르면 자체 모델 서빙. 성공 기준은
`eval_local.py blindguide-rewriter v1`의 선천맹 색 누출이 v1/v2 대비 실제로 줄어드는 것.

### 9.4 배포 발견 — Ollama Modelfile로 v2를 구우면 퇴화한다 (검증)

v2를 단일 태그로 굽으려 Modelfile에 SYSTEM/MESSAGE로 구웠더니 **temp0 greedy에서 퇴화**했다:
MESSAGE few-shot은 새 입력에서 예시 답을 그대로 되뱉고, 긴 SYSTEM은 입력에 없는 사물을
환각하고 메타 잡담을 붙였다. **똑같은 텍스트를 호출 시 system/user 메시지로 넘기면 정상**이며
eval의 v2 수치도 그 경로다. → v2는 굽지 않고 `model/serve_v2.py`처럼 **호출 시 프롬프트 전달**을
정본으로 한다. 단일 태그 굽기는 짧은 원본 프롬프트만 쓰는 **파인튜닝 모델**의 몫으로 남긴다
(짧은 SYSTEM은 이 퇴화를 안 일으킴). 이 발견이 배포 설계를 '호출 시 전달' + '파인튜닝 태그'로
가른 근거다.

### 9.5 완성 상태

1~3단계(로컬 구동·평가셋·프롬프트 최대화)는 데스크탑에서 실측 완료. 4~6단계(증류 데이터·LoRA·
서빙)는 데이터 생성·형식화까지 실증하고, 학습은 GPU 박스에서 바로 돌 스크립트로 완비. 남은
사람 몫: (a) train.review.jsonl 감수→train.jsonl 확정, (b) 검토 창의 베이스 모델·파인튜닝 착수
결정, (c) 메가존에서 train_lora.py 실행. 그 뒤 backend는 모델 태그 한 줄 교체.
